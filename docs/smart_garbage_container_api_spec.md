# Smart Underground Garbage Container — Backend API Specification

Version: `1.0.0`  
Target backend: REST API + MongoDB Atlas  
Target device: Arduino UNO Q edge device  
Device camera AI states: `EVERYTHING_OK` / `GARBAGE_DETECTED`

---

## 1. Purpose

This document defines the API contract between:

1. **Smart garbage container edge devices** running on Arduino UNO Q.
2. **Backend server** responsible for telemetry ingestion, event storage, device configuration, dashboard state, route planning, and maintenance.
3. **Web/admin frontend** used by operators, dispatchers, and maintenance teams.

The API must allow many garbage containers to connect, send telemetry, report AI camera detections, upload event images, and receive configuration from the backend.

---

## 2. System overview

Each container device sends:

- Environmental telemetry: temperature, humidity, light.
- Fill telemetry: ultrasonic distance, load-cell weight, calculated fill percentage.
- Camera AI state: `EVERYTHING_OK` or `GARBAGE_DETECTED`.
- Device health: uptime, network signal, firmware versions, offline queue count.
- Events: garbage detected, garbage cleared, full threshold, critical full, tamper, sensor fault, camera fault.

The backend stores:

- Time-series telemetry in MongoDB Atlas.
- Latest container state for dashboard/map use.
- Events and event lifecycle.
- Optional event images.
- Maintenance tickets.
- Route plans for garbage collection.

## 4. Core design rules

### 4.1 Transport

Use HTTPS for production.

For local development, HTTP is acceptable.

### 4.2 Encoding

All request and response bodies use JSON unless explicitly stated otherwise.

```http
Content-Type: application/json
Accept: application/json
```

### 4.3 Time format

All timestamps must use ISO 8601 UTC format.

Example:

```text
2026-04-25T01:25:00.000Z
```

The backend should reject timestamps that are not parseable.

### 4.4 Units

Use these units everywhere:

| Field | Unit |
|---|---|
| temperature_c | degrees Celsius |
| humidity_pct | percent, 0–100 |
| light_lux | lux |
| ultrasonic_distance_cm | centimeters |
| weight_kg | kilograms |
| fill percentages | percent, 0–100 |
| confidence | float, 0.0–1.0 |
| rssi_dbm | dBm |
| uptime_sec | seconds |
| free_disk_mb | megabytes |

### 4.5 Device idempotency

Every device message must include a unique `message_id`.

The backend must deduplicate messages by:

```text
device_id + message_id
```

If the same message is received twice, the backend must return success and must not create duplicate telemetry/events.

### 4.6 Sequence numbers

Every device should send an increasing integer `seq`.

The backend should store it but must not rely on strict ordering, because offline buffering can cause delayed messages.

### 4.7 Offline buffering

The device may be offline. When it reconnects, it can send old telemetry batches.

Backend must accept readings where `reading.ts` is older than `sent_at`.

### 4.8 Schema version

Every device request must include:

```json
"schema_version": "1.0"
```

Backend should reject unsupported major versions.

## 5. Authentication

### 5.1 MVP authentication

For the MVP, use a device token.

Device requests include:

```http
Authorization: DeviceToken <device_token>
X-Device-Id: cont-000123
```

The backend validates that:

1. The token exists.
2. The token belongs to the `X-Device-Id` device.
3. The device is active.

### 5.2 Future production authentication

Later, replace or extend the token with HMAC signing.

Suggested headers:

```http
X-Device-Id: cont-000123
X-Timestamp: 2026-04-25T01:25:00.000Z
X-Nonce: 8f63b0ec-7d0a-4e2d-8d9e-bef29c98a121
X-Signature: <base64-hmac-sha256>
```

Signature base string:

```text
METHOD + "\n" +
PATH + "\n" +
X-Timestamp + "\n" +
X-Nonce + "\n" +
sha256(request_body)
```

## 6. Common device message fields

All device-originated JSON messages must include these fields:

```json
{
  "schema_version": "1.0",
  "message_id": "2de3c0e6-463d-4f7e-8bd8-27fcb00bb8aa",
  "device_id": "cont-000123",
  "container_id": "bin-bg-001",
  "sent_at": "2026-04-25T01:25:00.000Z",
  "seq": 1042
}
```

Field definitions:

| Field | Type | Required | Description |
|---|---:|---:|---|
| schema_version | string | yes | API payload schema version. MVP uses `1.0`. |
| message_id | string UUID | yes | Unique message ID generated by the device. Used for idempotency. |
| device_id | string | yes | Unique edge device ID. |
| container_id | string | yes | Physical garbage container ID assigned by backend. |
| sent_at | string datetime | yes | Time when device sent this request. |
| seq | integer | yes | Monotonic sequence number from device. |

## 7. Enumerations

### 7.1 Camera state

```text
EVERYTHING_OK
GARBAGE_DETECTED
UNKNOWN
CAMERA_FAULT
```

For normal operation, the device should send only:

```text
EVERYTHING_OK
GARBAGE_DETECTED
```

`UNKNOWN` may be used after boot before the first valid inference.

`CAMERA_FAULT` may be used if the camera is not working.

### 7.2 Fill state

```text
EMPTY
NORMAL
NEAR_FULL
FULL
CRITICAL
UNKNOWN
```

Recommended thresholds:

| State | Condition |
|---|---|
| EMPTY | `fused_fill_pct < 20` |
| NORMAL | `20 <= fused_fill_pct < 70` |
| NEAR_FULL | `70 <= fused_fill_pct < 85` |
| FULL | `85 <= fused_fill_pct < 95` |
| CRITICAL | `fused_fill_pct >= 95` |

### 7.3 Device status

```text
ONLINE
OFFLINE
DEGRADED
FAULT
MAINTENANCE
UNKNOWN
```

### 7.4 Event type

```text
GARBAGE_DETECTED
GARBAGE_CLEARED
FULL_THRESHOLD
CRITICAL_FULL
TAMPER_OPEN
TAMPER_CLOSED
SENSOR_FAULT
CAMERA_FAULT
LOW_POWER
DEVICE_OFFLINE
DEVICE_ONLINE
MAINTENANCE_REQUIRED
COLLECTION_CONFIRMED
```

### 7.5 Event severity

```text
INFO
WARNING
CRITICAL
```

Recommended mapping:

| Event | Severity |
|---|---|
| GARBAGE_DETECTED | WARNING |
| GARBAGE_CLEARED | INFO |
| FULL_THRESHOLD | WARNING |
| CRITICAL_FULL | CRITICAL |
| TAMPER_OPEN | WARNING or CRITICAL |
| SENSOR_FAULT | WARNING |
| CAMERA_FAULT | WARNING |
| LOW_POWER | WARNING |
| DEVICE_OFFLINE | WARNING |
| MAINTENANCE_REQUIRED | WARNING |
| COLLECTION_CONFIRMED | INFO |

### 7.6 Event status

```text
OPEN
ACKNOWLEDGED
RESOLVED
IGNORED
```


```json
{
  "schema_version": "1.0",
  "message_id": "2de3c0e6-463d-4f7e-8bd8-27fcb00bb8aa",
  "device_id": "cont-000123",
  "container_id": "bin-bg-001",
  "sent_at": "2026-04-25T01:25:00.000Z",
  "seq": 1042,
  "readings": [
    {
      "ts": "2026-04-25T01:24:00.000Z",
      "sensors": {
        "temperature_c": 21.6,
        "humidity_pct": 48.2,
        "light_lux": 124.0,
        "ultrasonic_distance_cm": 43.8,
        "weight_kg": 126.4,
        "tamper_open": false
      },
      "fill": {
        "height_pct": 72,
        "weight_pct": 64,
        "fused_pct": 72,
        "state": "NEAR_FULL",
        "confidence": 0.82
      },
      "vision": {
        "model_id": "garbage-classifier-v1",
        "camera_state": "EVERYTHING_OK",
        "confidence": 0.94,
        "last_inference_at": "2026-04-25T01:23:58.000Z"
      },
      "health": {
        "device_status": "ONLINE",
        "rssi_dbm": -61,
        "uptime_sec": 35321,

        "free_disk_mb": 2300,

        "sensor_faults": [],
        "camera_fault": false

```

### Field definitions

#### `readings[]`

| Field | Type | Required | Description |
|---|---:|---:|---|
| ts | datetime string | yes | Time when the reading was measured. |
| sensors | object | yes | Raw sensor values. |
| fill | object | yes | Calculated fill state. |
| vision | object | yes | Latest camera AI state. |
| health | object | yes | Device health values. |

#### `sensors`

| Field | Type | Required | Description |
|---|---:|---:|---|
| temperature_c | number | no | Temperature in Celsius. |
| humidity_pct | number | no | Relative humidity. |
| light_lux | number | no | Ambient light level. |
| ultrasonic_distance_cm | number | no | Distance from ultrasonic sensor to garbage surface. |
| weight_kg | number | no | Measured weight. |
| tamper_open | boolean | no | Whether enclosure/service door is open. |

#### `fill`

| Field | Type | Required | Description |
|---|---:|---:|---|
| height_pct | number | yes | Fill estimated from ultrasonic sensor. |
| weight_pct | number | yes | Fill estimated from load cell. |
| fused_pct | number | yes | Final fill estimate used by backend. |
| state | string | yes | One of `EMPTY`, `NORMAL`, `NEAR_FULL`, `FULL`, `CRITICAL`, `UNKNOWN`. |
| confidence | number | yes | Confidence 0.0–1.0. Lower if sensors disagree. |

#### `vision`

| Field | Type | Required | Description |
|---|---:|---:|---|
| model_id | string | yes | AI model version/name. |
| camera_state | string | yes | `EVERYTHING_OK`, `GARBAGE_DETECTED`, `UNKNOWN`, or `CAMERA_FAULT`. |
| confidence | number | yes | Model confidence 0.0–1.0. |
| last_inference_at | datetime string | yes | Timestamp of last camera inference. |

```json

  "message_id": "2de3c0e6-463d-4f7e-8bd8-27fcb00bb8aa",

  }
```
```json
}
```

## 8.3 Upload device event

The device calls this when a state change or important condition occurs.

```http
POST /api/v1/device/events
```

### Request: garbage detected

```json
{
  "schema_version": "1.0",
  "message_id": "b1241dd0-cf9b-4ab0-a6e0-2a94316c3e7a",
  "device_id": "cont-000123",
  "container_id": "bin-bg-001",
  "sent_at": "2026-04-25T01:25:00.000Z",
  "seq": 1043,
  "event": {
    "type": "GARBAGE_DETECTED",
    "severity": "WARNING",
    "started_at": "2026-04-25T01:24:30.000Z",
    "ended_at": null,
    "confidence": 0.87,
    "summary": "Garbage detected around the container.",
    "state": {
      "camera_state": "GARBAGE_DETECTED",
      "fused_fill_pct": 72,
      "fill_state": "NEAR_FULL"
    },
    "evidence": {
      "image_available": true,
      "local_image_id": "img-20260425-012430"
    }
  }
}
```

### Request: garbage cleared

```json
{
  "schema_version": "1.0",
  "message_id": "aa508101-1f54-47d2-92a9-4c7a72fca111",
  "device_id": "cont-000123",
  "container_id": "bin-bg-001",
  "sent_at": "2026-04-25T02:10:00.000Z",
  "seq": 1101,
  "event": {
    "type": "GARBAGE_CLEARED",
    "severity": "INFO",
    "started_at": "2026-04-25T02:09:45.000Z",
    "ended_at": null,
    "confidence": 0.91,
    "summary": "Camera state returned to everything OK.",
    "state": {
      "camera_state": "EVERYTHING_OK",
      "fused_fill_pct": 68,
      "fill_state": "NORMAL"
    },
    "evidence": {
      "image_available": false,
      "local_image_id": null
    }
  }
}
```

### Event field definitions

| Field | Type | Required | Description |
|---|---:|---:|---|
| type | string | yes | Event type enum. |
| severity | string | yes | `INFO`, `WARNING`, or `CRITICAL`. |
| started_at | datetime string | yes | When the event started. |
| ended_at | datetime string/null | no | Optional end time. |
| confidence | number | no | Confidence 0.0–1.0. |
| summary | string | yes | Human-readable event summary. |
| state | object | yes | Relevant state snapshot. |
| evidence | object | no | Optional image/evidence metadata. |

### Response: `201 Created`

```json
{
  "accepted": true,
  "event_id": "evt-661c2f",
  "upload_image": true,
  "media_upload_url": "/api/v1/device/events/evt-661c2f/media",
  "server_time": "2026-04-25T01:25:01.000Z"
}
```

If no image is needed:

```json
{
  "accepted": true,
  "event_id": "evt-661c2f",
  "upload_image": false,
  "media_upload_url": null,
  "server_time": "2026-04-25T01:25:01.000Z"
}
```

### Backend behavior

For `GARBAGE_DETECTED`:

- Create or update an open event.
- Mark container latest camera state as `GARBAGE_DETECTED`.
- Request image upload if `evidence.image_available = true`.

For `GARBAGE_CLEARED`:

- Resolve existing open `GARBAGE_DETECTED` event for that container.
- Mark camera state as `EVERYTHING_OK`.

For duplicate `message_id`:

- Return the same `event_id` if possible.
- Do not create duplicate events.

## 8.4 Upload event image

Used only when backend asks for an image after an event.

```http
POST /api/v1/device/events/{event_id}/media
```

### Request headers

```http
Authorization: DeviceToken <device_token>
X-Device-Id: cont-000123
Content-Type: multipart/form-data
```

### Multipart fields

| Field | Type | Required | Description |
|---|---:|---:|---|
| image | file | yes | JPEG image. |
| metadata | JSON string | yes | Metadata about image. |

### Metadata JSON

```json
{
  "device_id": "cont-000123",
  "container_id": "bin-bg-001",
  "captured_at": "2026-04-25T01:24:30.000Z",
  "width": 1280,
  "height": 720,
  "model_id": "garbage-classifier-v1",
  "camera_state": "GARBAGE_DETECTED",
  "confidence": 0.87,
  "privacy_processed": false
}
```

### Response: `201 Created`

```json
{
  "accepted": true,
  "media_id": "media-123",
  "event_id": "evt-661c2f",
  "stored": true
}
```

### Backend behavior

Backend should store:

- Binary image in object storage or filesystem.
- Metadata in MongoDB `media` collection.
- Reference from event to media ID.

For hackathon, storing images on local disk and metadata in MongoDB is acceptable.

# 13. Route planning endpoints

The route planner is responsible for deciding which containers should be visited and in what order.

The backend may implement this directly or call a separate route optimizer service.

Recommended algorithm library: Google OR-Tools.

---

## 13.1 Create route plan

```http
POST /api/v1/routes/plan
```

### Request

```json
{
  "date": "2026-04-25",
  "depot": {
    "lat": 44.8201,
    "lng": 20.4600,
    "name": "Central depot"
  },
  "vehicle_ids": ["truck-01", "truck-02"],
  "constraints": {
    "max_route_duration_min": 480,
    "include_threshold_pct": 70,
    "force_include_event_types": [
      "GARBAGE_DETECTED",
      "CRITICAL_FULL",
      "FULL_THRESHOLD"
    ],
    "allow_drop_low_priority": true
  }
}
```

### Backend candidate selection

Backend should include containers when any of these is true:

- `fused_fill_pct >= include_threshold_pct`
- open event type is in `force_include_event_types`
- container has not been collected for too many days
- container has critical maintenance/tamper status

### Priority score suggestion

```text
priority_score =
  0.50 * fused_fill_pct
+  20   * garbage_detected_open_event
+  20   * critical_full_event
+  10   * full_threshold_event
+  5    * days_since_collection_score
```

Where boolean event values are 0 or 1.

### Response

```json
{
  "route_plan_id": "rp-20260425-001",
  "status": "PLANNED",
  "summary": {
    "vehicles_used": 2,
    "stops": 34,
    "estimated_distance_km": 82.4,
    "estimated_duration_min": 390,
    "dropped_low_priority_stops": 4
  },
  "routes": [
    {
      "vehicle_id": "truck-01",
      "estimated_distance_km": 41.2,
      "estimated_duration_min": 195,
      "stops": [
        {
          "stop_id": "stop-001",
          "order": 1,
          "container_id": "bin-bg-001",
          "eta": "2026-04-25T06:18:00.000Z",
          "service_time_min": 8,
          "priority_score": 92,
          "reason": ["NEAR_FULL", "GARBAGE_DETECTED"]
        }
      ]
    }
  ]
}
```

5. Wait for device telemetry to confirm fill percentage dropped.

# 16. Backend alert rules

The backend should generate or update events based on telemetry.

## 16.1 Fill alerts

```text
IF fused_fill_pct >= 85 for 2 consecutive telemetry uploads
THEN create FULL_THRESHOLD event.
```

```text
IF fused_fill_pct >= 95 for 1 telemetry upload
THEN create CRITICAL_FULL event.
```

## 16.2 Camera alerts

The device already sends `GARBAGE_DETECTED` events, but backend should also update the latest state from telemetry.

```text
IF telemetry.vision.camera_state = GARBAGE_DETECTED
THEN latest_state.camera_state = GARBAGE_DETECTED.
```

```text
IF device sends GARBAGE_CLEARED
THEN resolve open GARBAGE_DETECTED event.
```

## 16.3 Offline alerts

```text
IF no heartbeat or telemetry from device for > 10 minutes
THEN mark device OFFLINE and create DEVICE_OFFLINE event.
```

```text
IF device reconnects after OFFLINE
THEN mark ONLINE and create DEVICE_ONLINE event.
```

## 16.4 Sensor fault alerts

```text
IF ultrasonic_distance_cm is constant for > 6 hours
AND weight_kg changes significantly
THEN create SENSOR_FAULT event.
```

```text
IF camera_state = CAMERA_FAULT
THEN create CAMERA_FAULT event.
```

# 17. Error format

All errors should follow this structure:

```json
{
  "error": {
    "code": "INVALID_REQUEST",
    "message": "Human-readable message.",
    "details": {
      "field": "readings"
    }
  }
}
```

Common error codes:

| HTTP | Code | Meaning |
|---:|---|---|
| 400 | INVALID_REQUEST | Malformed JSON or missing required field. |
| 400 | INVALID_SCHEMA_VERSION | Unsupported schema version. |
| 400 | INVALID_TELEMETRY | Telemetry validation failed. |
| 401 | INVALID_DEVICE_TOKEN | Device token missing/invalid. |
| 403 | DEVICE_DISABLED | Device exists but is disabled. |
| 404 | DEVICE_NOT_FOUND | Device ID not found. |
| 404 | CONTAINER_NOT_FOUND | Container ID not found. |
| 409 | DEVICE_ALREADY_CLAIMED | Bootstrap conflict. |
| 413 | PAYLOAD_TOO_LARGE | Image or JSON payload too large. |
| 429 | RATE_LIMITED | Too many requests. |
| 500 | INTERNAL_ERROR | Unexpected backend error. |

# 18. Rate limits and payload limits

Recommended MVP limits:

| Request | Limit |
|---|---:|
| Telemetry batch size | 100 readings max |
| Telemetry JSON body | 256 KB max |
| Event JSON body | 64 KB max |
| Image upload | 2 MB max |
| Heartbeat interval | minimum 30 sec |
| Telemetry interval | minimum 10 sec |

If a device sends too much data:

```http
429 Too Many Requests
```

or

```http
413 Payload Too Large
```

# 22. Notes for embedded developer

The embedded application should implement these behaviors:

1. Keep a local queue of all unsent messages.
2. Retry failed messages with the same `message_id`.
3. Never send a new garbage event every frame.
4. Send `GARBAGE_DETECTED` only when state changes from OK to detected.
5. Send `GARBAGE_CLEARED` only when state changes from detected to OK.
6. Upload image only if backend response says `upload_image: true`.
7. Continue sending telemetry even if image upload fails.
8. Include firmware/model versions in heartbeat.
9. Pull config periodically or when backend response shows a new `config_revision`.
10. Use UTC timestamps if possible. If device time is unknown, still send readings but mark time sync status in health in a future version.

---

# 23. Notes for backend developer

The backend should implement these behaviors:

1. Device endpoints must be idempotent by `device_id + message_id`.
2. Telemetry must update both time-series history and `containers.latest_state`.
3. Events must update event history and latest state.
4. `GARBAGE_CLEARED` should resolve open `GARBAGE_DETECTED` events for the same container.
5. Image upload is optional and should not block telemetry ingestion.
6. Offline devices should be detected by scheduled job using `last_seen_at`.
7. All user/admin actions should be auditable later.
8. MongoDB indexes should be created before loading many readings.
9. Dashboard should mostly read from `containers.latest_state`, not scan telemetry history.
10. Historical charts should query `telemetry_timeseries`.
